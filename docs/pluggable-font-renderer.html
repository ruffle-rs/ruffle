This document describes the existing Ruffle device-font rendering pipeline, the additive integration that lets external renderers be plugged in at runtime, the contract those renderers must honour, and the two reference implementations developed alongside the contract: gdi-font-renderer (Windows GDI via napi-rs) and swash-font-renderer (pure-WebAssembly via swash).

1Ruffle font rendering architecture, before and after

Ruffle's text pipeline is split into three layers that each own a different concern: the SWF interpreter and ActionScript layout in ruffle_core, a font abstraction that hides the difference between embedded (SWF-supplied) fonts and host ("device") fonts in ruffle_core::font, and a per-platform UiBackend implementation that decides where unknown device fonts come from. On the web, that backend is WebUiBackend.

This three-layer split is pre-existing, and this branch does not change it. What does change is the set of rasterizers that WebUiBackend can dispatch to. On mainline Ruffle there are two: Embedded (bitmaps from SWF-provided outlines or from TTFs passed via fontSources) and Canvas (OffscreenCanvas.measureText/fillText). This branch adds a third variant, Custom, that routes every per-glyph call to a JS-side bridge object installed by the embedding host on globalThis.__ruffleCustomFontRenderer — the pluggable renderer that gives this document its name. The branch also teaches the shared glyph cache how to accommodate pixel-locked backends that cannot reuse a single bitmap across sizes.

1.1 Glossary

1.2 Pipeline overview

Common to both mainline and this branch: when the AVM needs to draw text, EditText in core/src/html walks the run of formatted text, asks the resident Font for each glyph at the requested height, and feeds the resulting GlyphRefs to the active render backend. If the requested font is unknown — no SWF tag defined it and no previous lookup cached it — the layout falls back to the Player's load_device_font() hook, which the web build delegates to WebUiBackend::load_device_font. That method inspects the chosen DeviceFontRenderer variant and registers a FontDefinition carrying the appropriate FontRenderer implementation. From that point on the layout loop treats the font like any other: metrics come from the trait, bitmaps from per-glyph calls, and the cache in ruffle_core::font::Font makes sure no glyph is rasterized twice.

What this branch changes (highlighted in the diagram): the FontRenderer trait gains a Custom arm that forwards per-glyph calls to the JS bridge; the ruffle_core::font::Font cache key grows from a single code point to a (code_point, size_px) tuple (size-unaware backends keep sharing one bucket via cache_size = 0); pixel-locked glyphs carry an optional intrinsic_scale so the layout can draw them 1:1 to device pixels; and new _at_size trait overloads let backends opt into size-aware rasterization. The top four boxes — the SWF interpreter, the EditText layout, the load_device_font hook, and the WebUiBackend dispatcher — are unchanged. The branch is deliberately additive: any path that does not ask for a Custom renderer produces the same bitmaps as mainline.

1.3 Cache strategy and intrinsic_scale

The per-glyph cache inside ruffle_core::font::Font is the pivot around which the whole pluggable renderer contract is designed. Ruffle already had a cache, but it was built on an assumption — one bitmap per code point is enough — that pixel-locked backends like GDI cannot honour.

1.3.1 The original size-agnostic cache

On mainline, the ExternalRenderer variant of GlyphSource stored two RefCell<HashMap>s:

// Before (mainline)
ExternalRenderer {
    glyph_cache:   RefCell<FnvHashMap<u16, Option<Glyph>>>,
    kerning_cache: RefCell<FnvHashMap<(u16, u16), Twips>>,
    font_renderer: Box<dyn FontRenderer>,
}

get_by_code_point(&self, code_point: char) took no size argument, and the trait had a single render_glyph(char) -> Option<Glyph>. The implicit contract: the renderer rasterizes once at a canonical em-size (64 px for the canvas backend), and the layout computes a runtime scale factor: 

// Before (mainline layout loop)
let scale = params.height.get() as f32 / resolution.font.scale();

Multiplying every glyph by scale happens on the GPU as part of the normal draw transform. The cache could therefore be keyed solely by code point, and memory stayed bounded regardless of how many sizes a movie used: one bitmap served them all. This worked because the canvas backend uses the browser's vector fillText pipeline (scaling a 64 px-em rasterization to a 12 px run is acceptable — the browser hints internally), and the embedded path feeds vector outlines to the GPU (scaled with no quality loss). Neither needed per-size storage.

1.3.2 Why this broke for pixel-locked backends

The defining property of a GDI-style backend is that the only way to get a crisp glyph is to ask the OS to rasterize it at the exact display size — with its own hinting, stem snapping, and native antialiasing (ClearType on Windows). The moment the layout scales such a bitmap by any factor other than 1.0, bilinear filtering smears the subpixel-tuned output into the blur the renderer was supposed to avoid. To keep GDI output pixel-perfect, the bitmap must reach the framebuffer with scale = 1.0. That requires two things the mainline cache did not provide:

  • A separate cache entry per display size, so a 12 px and a 24 px render of the same code point do not collide on the same key.
  • A way for a bitmap to carry its own em-scale, so the layout divisor cancels and the runtime scale collapses to 1.0 rather than stretching the bitmap.

A naive fix — always key by (code_point, size_px) — would have solved the GDI case at a substantial memory cost for canvas and embedded: the same glyph re-stored for every size, even though the bytes are identical modulo GPU scaling.

1.3.3 The size-aware rework, free for existing backends

The redesign splits "how to key the cache" away from the cache itself, and lets each backend opt in. Three changes to the trait:

pub trait FontRenderer: std::fmt::Debug {
    // Unchanged size-agnostic hooks (used by canvas / embedded):
    fn render_glyph(&self, character: char) -> Option<Glyph>;
    fn calculate_kerning(&self, left: char, right: char) -> Twips;
    // New: opt-in size-aware variants, default-delegate.
    fn render_glyph_at_size(&self, character: char, _height_px: u32)
        -> Option<Glyph> { self.render_glyph(character) }
    fn calculate_kerning_at_size(&self, left: char, right: char,
                                 _height_px: u32) -> Twips {
        self.calculate_kerning(left, right)
    }
    // New: tells the cache whether to key by size.
    fn is_size_aware(&self) -> bool { false }
}

The cache then extends its key tuples with a height_px component and picks the effective key at lookup time based on the backend's own flag:

// After (this branch)
ExternalRenderer {
    glyph_cache:   RefCell<FnvHashMap<(u16, u32), Option<Glyph>>>,
    kerning_cache: RefCell<FnvHashMap<(u16, u16, u32), Twips>>,
    font_renderer: Box<dyn FontRenderer>,
}

let cache_size = if font_renderer.is_size_aware() {
    height_px   // one entry per display size
} else {
    0           // every size collapses onto the same bucket
};
glyph_cache.borrow_mut()
    .entry((code_point, cache_size))
    .or_insert_with(|| font_renderer.render_glyph_at_size(ch, cache_size));

The cache_size = 0 branch is the whole point: size-unaware backends keep using a single bucket per code point, so canvas and embedded fonts pay zero extra memory after the rework — the only change for them is the extra u32 in the key and an unused _height_px argument the defaults discard. Two auxiliary pieces make it end-to-end:

  • Twips-to-pixel quantization. A helper twips_to_px_for_cache(Twips) -> u32 converts heights with half-pixel rounding ((twips + 10) / 20, clamped to ≥ 1). Without it, floating-point noise would fragment the cache into near-duplicate entries at 11.97 / 12.00 / 12.03 px.
  • Height plumbed through the lookup APIs. Font::get_glyph_for_char, GlyphSource::get_by_code_point, FontLike::resolve_glyph and get_kerning_offset all gained a height: Twips parameter, carried by the layout loop via params.height. Existence-only probes (has_glyphs_for_str) pass Twips::ZERO, which canonicalizes to the shared cache_size = 0 slot.

1.3.4 Glyph::intrinsic_scale and the 1:1 layout path

Cache keying alone is not enough: the layout also needs to stop rescaling the bitmap. Glyph gained an intrinsic_scale: Option<f32> field and a constructor from_bitmap_at_scale used by pixel-locked backends:

pub struct Glyph {
    shape: GlyphShape,
    advance: Twips,
    character: char,
    /// Em-size (in twips) at which this bitmap was rasterized.
    /// None falls back to the owning font's canonical scale.
    intrinsic_scale: Option<f32>,
}

The layout loop then consults it:

// After (this branch)
let glyph_scale_basis = glyph
    .intrinsic_scale()
    .unwrap_or_else(|| resolution.font.scale());
let scale = params.height.get() as f32 / glyph_scale_basis;

When a backend rasterizes at the display size and sets intrinsic_scale = params.height, the divisor cancels and scale collapses to 1.0 — every bitmap pixel lands on exactly one device pixel, with no bilinear blur. Backends that do not set the field (canvas, embedded) keep using the font's canonical scale exactly as before. Net effect: two independent mechanisms — size-aware cache keying and intrinsic_scale — each a pure addition guarded by a default that preserves the old behavior, each solving half of the GDI problem.

2Integrating a custom renderer into Ruffle

The integration is deliberately additive: a single new variant is added to the existing DeviceFontRenderer enum, and the rest is the minimum plumbing needed to (a) carry that variant from JS configuration to the Rust UI backend and (b) bridge per-glyph calls across the JS/WASM boundary. No previous code path was removed. All paths below are relative to the repository root.

2.1 core/src/font.rs — extend the FontRenderer trait

Two responsibilities were grafted onto the trait: size-aware rasterization, and a way for backends to tag bitmaps with the em-scale they chose. The defaults are no-ops, so the canvas renderer is unaffected.

pub trait FontRenderer: std::fmt::Debug {
    fn scale(&self) -> f32;
    fn get_font_metrics(&self) -> FontMetrics;
    fn has_kerning_info(&self) -> bool;
    fn render_glyph(&self, character: char) -> Option<Glyph>;
    fn calculate_kerning(&self, left: char, right: char) -> Twips;
    fn render_glyph_at_size(&self, character: char, _height_px: u32)
        -> Option<Glyph> { self.render_glyph(character) }
    fn calculate_kerning_at_size(&self, left: char, right: char,
                                 _height_px: u32) -> Twips {
        self.calculate_kerning(left, right)
    }
    fn is_size_aware(&self) -> bool { false }
}

The Glyph struct also gained intrinsic_scale: Option<f32> and a constructor that sets it (from_bitmap_at_scale); Font::evaluate_layout_operation consults it so that a backend rasterizing at the display size produces a glyph whose em-scale equals params.height, the divisor cancels, and scale collapses to 1.0 (no bilinear blur).

2.2 web/src/lib.rs — new DeviceFontRenderer variant

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum DeviceFontRenderer {
    Embedded,
    Canvas,
    /// Delegates glyph rasterization to a pluggable bridge
    /// object installed on `globalThis.__ruffleCustomFontRenderer`.
    /// Fetched lazily on every font request.
    Custom,
}

Why a unit variant: it keeps RuffleInstanceBuilder clonable without carrying any JS-side state. The bridge lives entirely in the JS global namespace and is read through reflection by WebUiBackend; so the same bridge can be shared across every player instance on the page, and bridges that load asynchronously (ES modules with top-level await) can be assigned after player construction without further plumbing.

2.3 web/src/builder.rs — recognise "custom"

#[wasm_bindgen(js_name = "setDeviceFontRenderer")]
pub fn set_device_font_renderer(&mut self, value: String) {
    self.device_font_renderer = match value.as_str() {
        "embedded" => DeviceFontRenderer::Embedded,
        "canvas"   => DeviceFontRenderer::Canvas,
        "custom"   => DeviceFontRenderer::Custom,
        _ => return,
    };
}
// Inside create_player:
if matches!(self.device_font_renderer, DeviceFontRenderer::Embedded) {
    self.setup_fonts(&mut core);
}

The match is intentionally tolerant — an unknown value keeps the previous default rather than panicking. The setup_fonts guard prevents Ruffle from registering its own TTF parses on top of the bridge: the bridge is the sole owner of the names it claims. There is no setter for the bridge itself — it lives on the JS global slot and is fetched by the UI backend, so the builder carries no per-instance state.

2.4 web/src/ui.rs — dispatch in load_device_font

WebUiBackend now stores the chosen DeviceFontRenderer and dispatches on it. The Custom arm looks up the bridge on the global slot at each call, then tries to instantiate the bridge-backed renderer — falling back to canvas if the slot is empty, the structural check fails, or the bridge rejects the family.

const CUSTOM_FONT_RENDERER_GLOBAL: &str = "__ruffleCustomFontRenderer";

fn lookup_custom_font_bridge() -> Option<JsValue> {
    let global = js_sys::global();
    let bridge =
        js_sys::Reflect::get(&global, &JsValue::from_str(CUSTOM_FONT_RENDERER_GLOBAL)).ok()?;
    if bridge.is_undefined() || bridge.is_null() { return None; }
    let create =
        js_sys::Reflect::get(&bridge, &JsValue::from_str("createRenderer")).ok()?;
    if !create.is_function() { return None; }
    Some(bridge)
}

fn load_device_font(&self, query: &FontQuery,
                    register: &mut dyn FnMut(FontDefinition)) {
    match &self.device_font_renderer {
        DeviceFontRenderer::Embedded => { /* registered up-front */ }
        DeviceFontRenderer::Canvas => register_canvas_font_renderer(query, register),
        DeviceFontRenderer::Custom => {
            let Some(bridge) = lookup_custom_font_bridge() else {
                tracing::warn!("...missing or invalid...");
                register_canvas_font_renderer(query, register);
                return;
            };
            match font_renderer::CustomFontRenderer::new(
                query.is_italic, query.is_bold, &query.name, &bridge,
            ) {
                Ok(r) => register(FontDefinition::ExternalRenderer { /* ... */ }),
                Err(e) => { tracing::warn!("..."); register_canvas_font_renderer(query, register); }
            }
        }
    }
}

Why lookup on every call: the bridge is produced and installed by the embedding host — Ruffle never loads an addon itself — so the slot may be empty at player construction and materialize later (e.g. a <script type="module"> with top-level await finishing after DOMContentLoaded). Reading the global on every font request means any bridge that arrives before the first glyph is picked up automatically. Any host condition can still leave the slot empty; the soft fallback keeps the experience at "some text quality, no broken page".

2.5 web/src/ui/font_renderer.rs — the bridge proxy

CustomFontRenderer is the new FontRenderer implementation. It receives the bridge as a &JsValue, invokes createRenderer(family, bold, italic) on it, caches the canonical-size metrics and the kerning-support flag, and forwards every per-glyph call via reflection (js_sys::Reflect). Any per-backend caching belongs to the addon, not to Ruffle.

  • Bridge plumbing: the constructor accepts the bridge as &JsValue (no global lookup inside the renderer), calls createRenderer once, and stores the returned JS object.
  • Metrics cache: getFontMetrics is invoked once at the canonical size (64 px) and stored as twips — steady-state operation never hits JS for metrics.
  • Per-glyph forwarding: renderGlyph(code, size_px) returns { width, height, bitmapTx, advance, pixels } or null; the proxy converts pixel fields to twips and produces Glyph::from_bitmap_at_scale(...) with intrinsic_scale = size_px * 20.
  • Kerning: routed through calculate_kerning_at_size; pixel values are converted back to twips on the Rust side, so the addon API stays in pixel units.
  • Lifecycle: Drop calls destroy() on the JS object, ignoring errors (best effort).

Why this shape: pushing caches down into the addon avoids growing Ruffle with addon-specific knowledge (HFONT pools, swash ScaleContexts, ttf-parser Faces). What stays in Rust is purely the JS plumbing.

2.6 web/packages/core/src/internal/custom-font-bridge.ts

This module owns the JS side of the contract: the FontBridge, CustomFontRenderer, FontMetrics and GlyphRaster interfaces plus helpers. There is no JS-side registry object: the bridge lives directly on globalThis.__ruffleCustomFontRenderer and is read on demand.

ExportRole
CUSTOM_FONT_RENDERER_GLOBALThe well-known name of the global slot, "__ruffleCustomFontRenderer". Kept in sync with the constant of the same name in web/src/ui.rs so TS and Rust never disagree on where to look.
isFontBridgeStructural check that a value is an object with a createRenderer function. Everything else in the contract is optional and probed later through reflection on the Rust side.
getCustomFontRendererReads globalThis.__ruffleCustomFontRenderer, runs it through isFontBridge, returns the bridge or undefined. Each call re-reads the global, so bridges assigned after player construction are picked up as soon as they materialize.

Why one global slot and not a registry: an earlier iteration kept a __ruffleFontBridges object keyed by module id so multiple addons could coexist. That bought nothing in practice — Ruffle only ever dispatches to one renderer per player — and made the contract harder to reason about. The final shape is deliberately minimal: one JS global, one structural check, one lazy getter. Producing the bridge object remains entirely the host's responsibility.

2.7 public/config/custom-font-bridge.ts and load-options.ts

The public @ruffle-rs/core entry point needed one new re-export module (consumer-facing types only — there is nothing for an addon to call on the Ruffle side) and a three-state enum.

// public/config/custom-font-bridge.ts
export {
    type FontBridge,
    type CustomFontRenderer,
    type FontMetrics,
    type GlyphRaster,
} from "../../internal/custom-font-bridge";

// public/config/load-options.ts
export enum DeviceFontRenderer {
    Embedded = "embedded",
    Canvas   = "canvas",
    Custom   = "custom",
}
deviceFontRenderer?: DeviceFontRenderer;

Because the bridge is identified by a fixed global slot — not a per-addon module id — the option collapses to a plain enum with three discrete states. An earlier design used a literal-template type `custom:${string}` to carry a module id; that field is gone because no such id exists anymore.

2.8 web/packages/core/src/internal/builder.ts

With the bridge living on a single global slot that may be populated asynchronously, the JS side no longer pre-validates the bridge — any such check would race the async loading it was meant to guard — and simply forwards the verbatim string to Rust. Validation and soft fallback happen later, at font-request time.

if (isExplicit(config.deviceFontRenderer)) {
    builder.setDeviceFontRenderer(config.deviceFontRenderer);
}

The previous design parsed a "custom:<id>" prefix and looked the id up in a JS registry, downgrading to canvas on a miss. That logic was removed with the registry; checking the global at this exact moment would produce false negatives for bridges that legitimately arrive a few microtasks later. Letting the Rust side own the lookup — once per font request, where the answer has to be right anyway — is simpler and more accurate.

2.9 web/packages/core/src/internal/player/inner.tsx

Ruffle already fetches every URL from fontSources at startup; the bytes were only pushed to the embedded path (builder.addFont). With Custom renderers, inner.tsx also forwards the same bytes to the bridge (when present and implementing registerFontData), so bridges without OS-font access can build their face index from the same blob with no extra fetch.

if (this.loadedConfig?.fontSources) {
    const isCustom =
        this.loadedConfig.deviceFontRenderer === DeviceFontRenderer.Custom;
    for (const url of this.loadedConfig.fontSources) {
        try {
            const response = await fetch(url);
            const bytes = new Uint8Array(await response.arrayBuffer());
            builder.addFont(url, bytes);
            if (isCustom) {
                // Lazy lookup: a bridge that finishes its async
                // init mid-fetch still gets the bytes fetched after it.
                const bridge = getCustomFontRenderer();
                if (bridge?.registerFontData) {
                    try { bridge.registerFontData(url, bytes); }
                    catch (err) {
                        console.warn(
                            `Ruffle: __ruffleCustomFontRenderer threw ` +
                            `from registerFontData("${url}")`, err,
                        );
                    }
                }
            }
        } catch (err) { /* fetch failure is logged */ }
    }
}

Why lazily in the loop: the bridge may be assigned by an async module that finishes loading while inner.tsx is still iterating. Capturing it once outside the loop would strand later font sources; re-reading the global inside each iteration closes that window. The alternative — having each bridge fetch the URLs itself — would duplicate the request and push CORS concerns onto the addon, so Ruffle stays the single I/O owner and the bridge merely receives bytes it has already paid for.

2.10 Cargo.toml — workspace

The new module crates live under font-renderers/ at the repository root but are not added to the workspace members list; each defines an empty [workspace] table in its own Cargo.toml so Cargo treats them as standalone workspaces. The motivation is twofold: their build targets differ from the main project (a custom cdylib on Windows for the napi addon, wasm32-unknown-unknown for swash) and their dependency closures pull in libraries the rest of Ruffle does not need.

3The pluggable module contract

Everything Ruffle asks of a custom font renderer is spelled out by two TypeScript interfaces, FontBridge and CustomFontRenderer. A bridge object is a plain JavaScript value assigned to the well-known global slot globalThis.__ruffleCustomFontRenderer; Ruffle reads it on every font request and invokes methods by reflection. No part of the contract prescribes how the bridge is built.

3.1 Technology is up to the module author

Ruffle is deliberately blind to the runtime nature of a bridge: web/src/ui.rs reads the global, confirms a createRenderer function, and hands the JsValue to the proxy, which forwards per-glyph calls via js_sys::Reflect. Whatever lives on the other side is opaque. The two reference modules — gdi-font-renderer (napi-rs native addon over the Windows GDI text stack) and swash-font-renderer (wasm-bindgen module rasterizing with pure-Rust swash) — are worked examples of two technologies, not exclusive choices. Other plausible implementations: a Core Text addon for macOS, a fontconfig-backed addon for Linux, a FreeType-based WASM module, a pure-JS Canvas 2D wrapper, or a test stub injecting deterministic bitmaps. Ruffle treats them all identically. The practical consequences of the technology choice fall entirely on the addon author and host: build pipeline, platform coverage, whether font bytes need feeding via registerFontData, and how the bridge ends up on the slot.

3.2 The shared FontBridge contract

interface FontBridge {
    createRenderer(
        family: string,
        bold: boolean,
        italic: boolean,
    ): CustomFontRenderer | null | undefined;
    // Optional. Implemented by bridges that need raw font bytes.
    registerFontData?(source: string, bytes: Uint8Array): void;
}

createRenderer is the only required method. Returning null/undefined tells Ruffle the bridge does not handle that family — the fallback to canvas is taken on the Rust side. Throwing is a hard error and surfaces as a one-time tracing warning. registerFontData is invoked once per fontSources entry, after the bytes are fetched, before the first createRenderer call. TTC/OTC collections are delivered in a single call — the bridge iterates the faces inside. Any exception is logged and swallowed; this method must never abort Ruffle's font-source processing.

3.3 The CustomFontRenderer instance contract

interface CustomFontRenderer {
    hasKerningInfo(): boolean;
    getFontMetrics(sizePx: number): FontMetrics;
    renderGlyph(codePoint: number, sizePx: number): GlyphRaster | null;
    calculateKerning(left: number, right: number, sizePx: number): number;
    destroy(): void;
}
interface FontMetrics { ascent: number; descent: number; leading: number; }
interface GlyphRaster {
    width: number;       // bitmap width, >= 1
    height: number;      // bitmap height, >= 1
    bitmapTx: number;    // x offset of bitmap from pen, may be negative
    advance: number;     // pen advance after this glyph
    pixels: Uint8Array;  // RGBA, row-major top-down, premultiplied
}

Conventions enforced by Ruffle:

  • All numeric values are in pixels at sizePx. Ruffle converts to twips on the Rust side — addons never see the twips coordinate system.
  • Bitmaps are RGBA, top-down, row-major, premultiplied alpha. Non-ink pixels are (0, 0, 0, 0). The buffer length must equal width * height * 4.
  • Bitmap layout uses the canvas convention: height = ascent + descent, baseline at row y = ascent. The result must already be padded vertically; Ruffle does not adjust it.
  • destroy() is called exactly once when Ruffle drops the renderer; the addon releases every native handle it owns. Calling it twice must be a no-op.

3.4 How a bridge is registered

One mechanism, regardless of what the addon is made of: the bridge object must sit on globalThis.__ruffleCustomFontRenderer before the first device-font glyph is requested. Ruffle performs no dynamic loading of its own — if the slot is empty or lacks a createRenderer function when a request arrives, load_device_font falls back to canvas for that request, logs a warning, and the page keeps rendering. The registration is a single assignment:

globalThis.__ruffleCustomFontRenderer = bridge;

Because the slot is read lazily, the assignment can happen before or after the player is constructed — what matters is that the bridge is in place by the time the first glyph has to be rasterized.

Example — Electron preload script (synchronous require):

// preload.js (runs with nodeIntegration available):
const bridge = require("@example/some-font-addon");
globalThis.__ruffleCustomFontRenderer = bridge;

Example — pure-browser ESM module (async import):

<script type="module">
  import init, * as bridge from "./some-font-addon.js";
  await init();
  globalThis.__ruffleCustomFontRenderer = bridge;
</script>

In the Electron snippet the assignment is synchronous and finishes before the page runs any scripts. In the ESM snippet it is asynchronous and may finish after the player is constructed — that is fine: the per-request lookup honours any bridge that lands before the first glyph. If it genuinely arrives too late (or never), Ruffle serves the affected fonts from canvas.

4The gdi-font-renderer module

gdi-font-renderer is one of the two reference renderers. It is a napi-rs native Node addon that delegates rasterization to the Windows GDI text stack, targeting Electron-based hosts on Windows where the goal is pixel-perfect, hinted output matching the native UI — the look the original Flash Player produced for device fonts. Ruffle does not depend on it and is unaware of its technology; what follows is an example of how a native-addon bridge is put together.

PlatformsWindows only. The addon calls directly into the Win32 GDI text stack, so only Windows targets compile. The napi CLI is invoked without --platform, emitting a single .node file with no architecture suffix — consumers require() it directly.

4.2 Architecture

A single napi-rs class plus a thin factory. The class lazily creates a per-pixel-size SizedFont, each owning one HFONT. Per-size caches share a single Mutex<Inner> guard so napi-rs threads don't race on the GDI device context.

pub struct GdiFontRenderer {
    family: String,
    bold: bool,
    italic: bool,
    inner: Mutex<Inner>,
}
struct Inner {
    /// HDC bound to the screen, used as the rasterization device.
    hdc: HDC,
    /// One SizedFont per pixel size requested by Ruffle.
    sizes: HashMap<u32, SizedFont>,
}
struct SizedFont {
    hfont: HFONT,
    metrics: GdiFontMetrics,
    /// Glyph cache keyed by code point.
    glyphs: HashMap<u32, Option<GdiGlyphRaster>>,
}

4.3 Rasterization details

  • Font handle: CreateFontW with lfHeight = -size_px (em-height in pixels), lfWeight FW_BOLD when bold, lfItalic = TRUE when italic. CLEARTYPE_QUALITY is requested.
  • Metrics: from GetTextMetricsW. tmAscent + tmDescent becomes the bitmap height; tmExternalLeading becomes FontMetrics.leading.
  • Glyph bitmap: GetGlyphOutlineW with GGO_GRAY8_BITMAP retrieves a 65-level gray ink mask plus a GLYPHMETRICS describing the black box and cell origin. The mask is expanded to RGBA premultiplied, alpha derived from the gray level.
  • Bitmap layout: padded vertically to height = ascent + descent, ink placed so the baseline lands at y = ascent. bitmapTx is the horizontal offset from the pen origin to the left edge of the ink box — usually 0 for ASCII, negative for left-overhanging italics.
  • Advance: GetCharABCWidthsW when available, falling back to GetCharWidth32W; in pixels.
  • Kerning: GetKerningPairsW queried lazily on first calculateKerning and hashed by (left, right). Absent pairs return 0.

4.4 hasKerningInfo policy

The implementation always returns true. Returning false would force Ruffle to skip calculateKerning for every pair, but pairs without a dedicated entry already return 0 — indistinguishable from "no kerning info" for that pair. Truthful self-reporting would require eagerly creating an HFONT and reading its kerning table; deferring to the first kerning request is cheaper.

5The swash-font-renderer module

swash-font-renderer is the other reference renderer: a wasm-bindgen WebAssembly module built around the swash rasterization library — a natural fit when the host is a regular browser and no OS font database is available. Font bytes are supplied through registerFontData, parsed once with ttf-parser, indexed by family/style, and rasterized on demand with hinted grayscale output.

5.2 Architecture

Single-threaded by construction: the WASM module runs on the main thread. State is kept in a thread_local! registry, avoiding the cost of a Mutex and matching the model wasm-bindgen offers.

#[derive(Clone, Eq, Hash, PartialEq)]

struct RegisteredFace {
    data: Rc<Vec<u8>>,    // shared font blob
    offset: u32,           // face offset (TTC/OTC support)
    has_kern_table: bool,

}

        = RefCell::new(HashMap::new());

5.3 registerFontData — face indexing

Called once per fontSources entry (§2.9). The implementation:

  • Calls ttf_parser::fonts_in_collection to figure out whether the blob is a single TTF/OTF or a TTC/OTC.
  • Iterates every face, parsing with ttf_parser::Face::parse(bytes, offset).
  • Pulls the family name from the name table (Unicode entries only), lowercases it, and combines it with the bold/italic flags into a FaceKey.
  • Stores a RegisteredFace sharing the underlying Rc<Vec<u8>> with all faces from the same blob — one heap allocation for the blob, regardless of face count.
  • Logs and returns silently on parse failures, matching the embedded path's graceful degradation.

createRenderer looks up the requested FaceKey. A miss returns None (JS null), which Ruffle reads as "the bridge does not handle this family" and falls back to canvas. A hit produces a fresh CustomFontRenderer with its own ScaleContext (swash's per-thread rasterization scratch) and an empty per-size cache. getFontMetrics(sizePx) populates by_size[size_px] on first call by querying MetricsProxy, scaling by size_px / units_per_em, and rounding.

5.5 renderGlyph — rasterization with bitmap padding

swash returns a tight bitmap whose origin is the glyph's vector origin (baseline, left side of the ink box). Ruffle expects a line-box bitmap where height = ascent + descent and the baseline sits at row y = ascent. The implementation pads the swash output vertically — this was the bug fix that resolved the early "text appears too high in the line box" issue.

fn render_glyph_entry(
    scale_context: &mut ScaleContext,
    font_ref: &FontRef<'_>,
    glyph_id: u16,
    ppem: f32,
    ascent_px: i32,
    descent_px: i32,
) -> Option<CachedGlyph> {
    let mut scaler = scale_context
        .builder(*font_ref)
        .size(ppem)
        .hint(true)
        .build();
    let image = Render::new(&[Source::Outline])
        .format(Format::Alpha)
        .render(&mut scaler, glyph_id)?;
    let placement = image.placement; // zeno::Placement, BottomLeft origin
    let in_w = placement.width as i32;
    let in_h = placement.height as i32;
    let out_w = in_w.max(1) as u32;
    let out_h = (ascent_px + descent_px).max(1) as u32;
    // Vertical padding: baseline sits at row `ascent_px`,
    // ink spans `placement.top` rows above the baseline.
    let dst_y_offset = ascent_px - placement.top;
    let mut pixels = vec![0u8; (out_w * out_h * 4) as usize];
    for src_y in 0..in_h {
        let dst_y = src_y + dst_y_offset;
        if dst_y < 0 || dst_y >= out_h as i32 { continue; }
        for src_x in 0..in_w {
            let alpha = image.data[(src_y * in_w + src_x) as usize];
            let i = ((dst_y as u32 * out_w + src_x as u32) * 4) as usize;
            // Premultiplied white
            pixels[i + 0] = alpha;
            pixels[i + 1] = alpha;
            pixels[i + 2] = alpha;
            pixels[i + 3] = alpha;
        }
    }
    Some(CachedGlyph {
        width: out_w, height: out_h,
        bitmap_tx: placement.left as f32,
        advance: scaler.advance(glyph_id).x,
        pixels,
    })
}

Quality settings: full hinting via .hint(true) and grayscale alpha (Format::Alpha) rather than subpixel — the result is portable across screens of any orientation/subpixel layout, which matches what a generic Flash player needs to look right on any setup.

5.6 calculateKerning

Kerning is read directly from the OpenType kern table via ttf-parser, iterating every subtable until a pair matches. The result, in font units, is scaled by size_px / units_per_em and returned in pixels. hasKerningInfo() honestly reports whether the parsed face has a kern table, so Ruffle can skip the per-pair calls when there is no point.

5.7 Lifecycle and memory

  • The Rc<Vec<u8>> blob is shared between the registry and every renderer derived from it; dropping a renderer alone does not free the blob.
  • destroy() is exposed but is essentially a no-op in WASM — the per-size cache is freed when the wasm-bindgen object is GC'd. It exists to satisfy contract symmetry with the napi-rs side.
  • Calling registerFontData twice for the same source is idempotent: identical FaceKeys overwrite the entry, letting the host re-register fonts without leaking the previous Rc.

5.8 Usage

The postbuild.mjs script rewrites the wasm-bindgen glue so the module self-initialises via top-level await. The bridge methods are plain named exports, so the whole module namespace object satisfies the FontBridge contract — a single await import(...) produces a value assignable to the slot as-is:

<script type="module">
  globalThis.__ruffleCustomFontRenderer =
      await import("swash-font-renderer.js");
</script>
<script src="ruffle.js"></script>
<script>
  window.RufflePlayer.config = {
      deviceFontRenderer: "custom",
      fontSources: ["tahoma.ttf", "tahomabd.ttf"],
  };
</script>

Script ordering here is convenient but not critical: the browser blocks the classic ruffle.js tag until the preceding module script finishes its top-level await, so the bridge is in place by the time the player is instantiated. Even if it were not, the per-request lookup in load_device_font would pick up a later bridge before the first glyph. Import failure is non-fatal: if the module script throws, the slot is never assigned and the first device-font request falls back to canvas with a logged warning. Wrapping the await import(...) in try/catch is advisable in production for diagnostics, but not required for correctness.

AAppendix — Future work: the JS / WASM boundary

Every pluggable renderer call — renderGlyph, calculateKerning, getFontMetrics — crosses the WASM/JavaScript boundary twice: Ruffle's WASM module calls into the bridge through js_sys::Reflect, the bridge dispatches to the backend, and the result is marshaled back. This is true whether the backend is a napi-rs native addon (GDI) or another WASM module (swash): both reach Ruffle through the same JS-side bridge object, so the marshaling cost is paid in both topologies.

A.1 What each boundary crossing costs

Per call, enough overhead to become visible in profiles when the cache is cold:

  • Copying primitive arguments (code point, size, bold/italic flags) through wasm-bindgen's generated glue.
  • For renderGlyph, allocating a JS object with five fields and a Uint8Array view of the pixel buffer.
  • Reading those fields back from Rust via Reflect::get, one call per field, with an Option<JsValue> round trip each time.
  • Copying the Uint8Array contents into a Vec<u8> so Rust can own the pixels.

The swash backend avoids the last step slightly — its pixels view aliases WASM linear memory directly — but the other four items are identical, including the reflective property lookups.

A.2 Why it is not a problem today

The Rust-side cache (§1.3) already amortizes the cost: every (code_point, height_px) bitmap crosses the boundary exactly once, when the layout first asks for it. Subsequent frames, retraced strings, and most re-layouts with the same runs hit the cache and never touch JavaScript. Metrics are cached at the canonical size on renderer creation; kerning is cached per pair per size. A typical movie in steady state is rendered with essentially no boundary traffic — it is the warm-up phase of a new font size that dominates, and that is bounded by the glyph set the movie actually uses.

A.3 When it might be worth optimizing

If profiling ever shows measurable boundary overhead — a frame-time spike when a long, glyph-diverse text field is first displayed, or a movie cycling through many sizes — these directions are worth considering, roughly in order of effort vs. benefit:

  • Batch glyph requests. A renderGlyphs(codes: Uint32Array, sizePx: number) entry point returning an array of descriptors in one JS call would collapse per-glyph overhead into per-string overhead.
  • Shared memory for pixel buffers. Rust preallocates a scratch ArrayBuffer sized to the largest expected glyph bitmap and hands it to the bridge for in-place writes — removing the Vec<u8> copy and the per-glyph Uint8Array allocation.
  • Bulk kerning queries. A getKerningTable(sizePx) hook returning the whole table once, indexed locally by Rust.
  • Flat struct ABI. Drop the object-return convention for a packed Int32Array whose known slots carry the metrics fields — no Reflect::get lookups, just array indexing. The object-based API would be kept as a compatibility mode.

Why deferred: each requires extending the public bridge contract and the TypeScript surface, which a third-party addon could not ignore without losing functionality. Introducing them speculatively would lock in an API shape before real workloads reveal the hot path. The cache-first design keeps the bridge narrow and the hot path cold; these should be revisited only once there is a concrete performance signal.
